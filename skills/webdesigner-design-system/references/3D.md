# Conditional 3D Workflow

This contract applies only when the user explicitly requests a 3D deliverable or when an existing product already contains a 3D experience that must be changed. It connects the project-level Nightglass interface system to the globally installed Mint production pipeline and the project-scoped Blender authored-DCC workflow.

Normal websites, dashboards, and application UI must not acquire decorative 3D merely because the capability is available.

## 1. Activation

Activate this workflow for an explicit request involving one or more of:

- A generated 3D model or coherent model asset pack.
- Rigging, character animation, retopology, or model optimization.
- PBR material generation, map extraction, or a material pack.
- A Three.js viewer, configurator, simulation, walkthrough, editor, game, or interactive scene.
- A generated Gaussian-splat world or environment.
- Audio, ambience, music, or sound effects for a 3D experience.
- Conversion or downloadable artifacts when the live Mint host confirms the requested format.

Do not activate it for:

- An ordinary landing page, dashboard, form, or content site.
- Decorative blobs, fake depth, card perspective, or a background object that can be handled by CSS, SVG, Canvas, or existing assets.
- A vague request for something dynamic, immersive, premium, or modern that does not explicitly require 3D.

When intent is ambiguous and adding 3D would materially change scope, cost, performance, or accessibility, ask before generating.

## 2. Required Capability Check

Before 3D work:

1. Choose the generated-asset, authored-DCC, or mixed route before calling a 3D tool.
2. For generated assets, read the installed mint-threejs-skills SKILL.md completely, read its route-specific references, and inspect the live Mint host.
3. For authored Blender work, read BLENDER.md and the relevant bundled specialist skill, then verify the local Blender host.
4. For mixed work, satisfy both routes and keep the original and derivative provenance separate.
5. Never invent tool names, request schemas, lifecycle stages, animation set IDs, downloadable formats, or conversion support.
6. If the selected capability is unavailable, report the exact blocker. Do not silently switch providers, enable a remote Blender integration, or ask for provider API keys.

Procedural Three.js geometry or user-provided assets remain valid when they are the better design choice or Mint lacks the required capability. Do not create a procedural or Blender duplicate of an asset Mint generated successfully.

## 3. Route Selection

### Authored or edited Blender deliverable

For precise geometry, topology, UVs, material tuning, armatures, weights, animation editing, inspection, repair, retopology, or format export:

- Read BLENDER.md.
- Use the relevant bundled Blender specialist skill.
- Work on a physical copy under `work/blender/<task>/`; never overwrite the selected source.
- Put standalone deliverables under `outputs/3d/<task>/`, unless an existing application's asset convention applies.
- Inspect scene, object, and viewport state before approval-gated arbitrary Blender Python.

### Asset as the deliverable

For one newly generated model, animated model, model pack, material, material pack, audio asset, or explicitly requested generated world:

- Read the skill suite's references/asset-viewer.md.
- Use the Three.js app director.
- Treat the asset as the product and any viewer as its inspection and download shell.
- Integrate into an existing app when one exists instead of scaffolding a separate viewer.

### General 3D application

For viewers, product configurators, simulations, walkthroughs, editors, portfolios, or interactive experiences:

- Use skills/threejs-app-director/SKILL.md.
- Add interaction, visual, debugging, and QA specialists only when required.

### Game or game-like experience

For objectives, challenge, scoring, failure, progression, or deliberate game feel:

- Use skills/threejs-game-director/SKILL.md.
- Add the gameplay, game UI, interaction, visual, debugging, and QA specialists selected by that director.
- Do not force game systems onto a general 3D app.

### Mixed experience

Start with the app director, then add only the relevant game specialists. Existing project architecture always wins. Mint owns requested generation; Blender owns requested authored DCC changes. Preserve the successful Mint original unless a user request or verified runtime requirement demands a separate derivative. Gaussian splats never move into the Blender route.

## 4. Architecture Defaults

- Preserve the current framework, build system, asset root, routing, and state model.
- For greenfield work, default to TypeScript, Vite, and Three.js modules.
- Vanilla Three.js is the default. Use React Three Fiber or another Three.js-based stack only when the project already uses it or the user requests it.
- Keep Mint and Blender MCP calls in agent tooling. Browser runtime code loads synchronized artifacts or approved remote world configuration; it never calls either MCP server.
- Do not create mint-assets.json until the first Mint artifact is actually integrated.
- Keep Blender derivatives distinct from Mint originals and their manifest entries. Do not create mint-assets.json for Blender-only work.

## 5. Generation Authorization

A direct user request authorizes one automatic Mint workflow for exactly the named deliverable. That authorization includes:

- The initial generation call for the requested asset or coherent pack.
- Required status waits and returned next steps.
- Automatic completion of the same requested run.
- Retrieval of its durable artifact manifest.
- Synchronization and integration of the requested result.
- Rigging, animation, retopology, optimization, or audio only when those operations were part of the user's original request.

Ask before:

- Retrying a failed or moderated generation.
- Revising a preview.
- Creating variants or additional assets.
- Expanding an asset pack beyond the requested item plan.
- Adding derivatives such as animation, retopology, or optimization that were not requested.
- Generating a world when the user requested only a scene, level, background, or environment concept.
- Any purchase, plan change, or other action outside available credits.

Use review mode only when the user asks to inspect, compare, revise, select, or approve previews. Otherwise use Mint's automatic mode.

## 6. Asset Production

### Plan first

Write a concise asset plan with:

- Deliverable and intended runtime role.
- Number of assets or explicit pack item list.
- Visual and material requirements.
- Scale, orientation, pivot, collision, animation, and interaction needs.
- Required output only if the live host supports it.

Prefer discrete generated models and compose them in Three.js. Use a generated world only for an explicit world or environment-level request.

### Follow live results

- Follow returned nextSteps instead of inventing a polling loop.
- Prefer the host wait tools and keep waits within their documented limits.
- Preserve real dimensions, aspect ratio, duration, byte size, extension metadata, and loader hints.
- Do not expose raw asset handles, provider identifiers, storage details, or internal failure data.
- For a retryable failure, summarize the safe reason and retry guidance, then request authorization before retrying.
- A moderation failure requires a changed prompt and renewed authorization.

### Durable project registry

For each imported Mint file or remote world configuration:

- Read references/asset-pipeline.md.
- Use scripts/sync-mint-assets.mjs from the installed suite.
- Maintain project-root mint-assets.json.
- Use stable human-readable logical keys and reuse them on updates.
- Preserve an existing project's asset-root conventions.
- For greenfield work without a convention, store ordinary synchronized files under public/assets/mint.
- Commit durable project paths and manifests, not temporary download URLs.

### Models and animation

- Read references/gltf-runtime-compatibility.md before loading a Mint GLB.
- Use the shared Draco-capable loader for models, animation clips, and colliders.
- Never load an optimized Mint GLB through a bare GLTFLoader.
- Preserve generated geometry, materials, textures, UVs, topology, and animation data unless the user explicitly requests a derivative.
- When a Blender derivative is required, preserve the Mint original and record the derivative, source logical key, reason, and output path separately.
- For a natural request containing several character behaviors, inspect the live curated animation-set catalog and bind returned semantic roles to runtime state. Never guess set IDs or reconstruct a set from remembered numeric actions.

### Materials

- Treat returned Base Color, Normal, Roughness, Metalness, and Height maps as the authored PBR set.
- Preserve exact files and color spaces.
- Do not invent missing maps, recolor Base Color, or replace a successful generated material with a runtime shader.

### Generated worlds

- Generate only after explicit world-level intent.
- Read references/mint-world-splats.md.
- Keep the returned RAD runtime and invisible collider under one shared transform.
- Do not replace the requested world with a collection of guessed procedural objects.

### Audio

- Preserve the returned duration, format, and artifact metadata.
- Load through the project's existing audio system when present.
- Provide a mute control and avoid autoplay with sound unless the user explicitly requests it and platform rules allow it.

## 7. Canvas-First Nightglass UI

The 3D scene is the dominant visual plane. Runtime UI exists to orient, control, inspect, recover, or complete the user's task.

### Composition

- Let the canvas fill the available working region.
- Use a compact bottom-centered control group for simple viewers and walkthroughs.
- Place loading, ready, and error status directly above that control group.
- Put secondary details in a compact centered dialog opened by an info button.
- Avoid permanent sidebars for a canonical asset viewer.
- Do not add a header, title bar, navigation, marketing copy, attribution, or decorative chrome unless the product requires it.

### Nightglass bridge

- Use the existing --ng-* design tokens for DOM controls and overlays.
- Use --ng-color-overlay for modal backdrops and --ng-color-surface-2 for floating controls.
- Use --ng-shadow-hairline before floating shadows.
- Use the aqua accent for the single primary action or active inspection mode.
- Keep controls at least --ng-control-height and preserve visible focus.
- Keep provider names, Mint branding, generation links, raw asset IDs, and provenance out of runtime UI unless explicitly requested.

### Interaction

- Provide clear pointer, keyboard, and touch controls.
- Make orbit, pan, zoom, selection, reset, pause, mute, and fullscreen behavior discoverable when present.
- Avoid interactions that depend exclusively on hover.
- Preserve user camera intent when opening UI and restore focus when dialogs close.
- Do not use scroll hijacking unless spatial navigation is the requested experience.

### Accessibility and fallback

- Give canvas experiences a meaningful accessible name and an adjacent text summary of the scene or task.
- Provide DOM equivalents for essential actions and information.
- Pair visual selection and status with labels or announcements.
- Respect reduced motion for UI, camera transitions, parallax, and nonessential ambient movement.
- Provide pause controls for continuous nonessential animation.
- Detect WebGL or asset-loading failure and show a recoverable Nightglass error state instead of a blank canvas.
- Do not make generated audio the only channel for important feedback.

### Runtime quality

- Bound device pixel ratio and adapt quality to the target device.
- Dispose replaced geometries, materials, textures, render targets, listeners, and animation mixers.
- Resize from the actual canvas container.
- Pause or reduce work when hidden or offscreen where the application permits it.
- Keep loading progress honest and avoid a false 100 percent state before scene readiness.

## 8. Verification Boundary

The installed skill's references/verification-policy.md overrides broader project browser and mobile gates for Mint-backed 3D work.

Blender-only or mixed DCC verification follows BLENDER.md. A read-only installation smoke test is limited to scene information and a viewport screenshot; it never authorizes Blender Python or scene modification.

### Automatic minimum

Without additional approval:

1. Run the target build, typecheck, or nearest compile gate.
2. Run focused existing non-browser tests for changed logic.
3. Confirm referenced project-local asset files and import paths resolve.
4. For Mint GLBs, inspect recorded extension metadata and confirm all model, animation, and collider paths use the shared Draco-capable loader.
5. Fix failures caused by the current work.

This automatic minimum does not authorize starting a preview server, opening or driving a browser, running Playwright, inspecting the canvas, capturing screenshots, profiling, visual regression, bot playtesting, or a viewport matrix.

### Approval-gated QA

- Ask before extended desktop/browser QA and name the proposed checks.
- Desktop approval does not include mobile QA.
- Ask separately before mobile QA.
- Do not claim browser-tested, fully playable, responsive, release-ready, or performance-verified behavior without the corresponding evidence.

## 9. Handoff

For meaningful Mint-backed work, report:

- Requested surface and output.
- Generation status and the Mint handoff link.
- Durable artifact or manifest and stable project path.
- mint-assets.json logical key and synchronization status.
- Runtime integration status.
- Verification that actually ran.
- Any exact capability, quota, moderation, format, or authentication blocker.

Mention provenance in the final response or developer documentation, not in the user's runtime interface.

For Blender-backed work, also report the source copy, working `.blend`, separately recorded derivatives, final exported paths, Blender version, and the exact scene, object, viewport, animation, and export checks that ran.
